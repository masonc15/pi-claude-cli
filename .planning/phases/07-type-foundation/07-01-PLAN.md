**Per D-02: Only replace `as any` casts with proper type narrowing. No behavioral changes to consuming code. The logical behavior of every code path must remain identical.**

**Fix 1: provider.ts line 238** -- `!(msg as any).parent_tool_use_id`

Replace with direct property access (now that `ClaudeStreamEventMessage` has `parent_tool_use_id`):

```typescript
// BEFORE:
const isTopLevel = !(msg as any).parent_tool_use_id;

// AFTER:
const isTopLevel = !msg.parent_tool_use_id;
```

This works because at line 238 we are inside `if (msg.type === "stream_event")`, so TypeScript narrows `msg` to `ClaudeStreamEventMessage`, which now has `parent_tool_use_id?: string | null`. The `!` operator handles both `undefined` (field absent) and `null` (explicitly top-level) the same as before.

**Fix 2: event-bridge.ts line 273** -- `(output.content[idx] as any).arguments = block.arguments`

Replace the `as any` with `as ToolCall` (the pi-ai type). This is inside the `if (block.type === "tool_use")` branch where `output.content[idx]` is known to be a `ToolCall`:

```typescript
// BEFORE:
(output.content[idx] as any).arguments = block.arguments;

// AFTER:
(output.content[idx] as ToolCall).arguments = block.arguments;
```

`ToolCall.arguments` is `Record<string, any>` and `block.arguments` is `Record<string, unknown>`. `Record<string, unknown>` is assignable to `Record<string, any>`.

**Fix 3: event-bridge.ts line 308** -- `delete (block as any).index`

With `TrackedContentBlock.index` now optional, the cast is no longer needed:

```typescript
// BEFORE:
delete (block as any).index;

// AFTER:
delete block.index;
```

Note: `block` is typed as `TrackedBlock` which is `TrackedContentBlock | TrackedToolBlock`. `TrackedToolBlock` has `index: number` (required). The `delete` is inside `handleContentBlockStop` which iterates over all block types. Since `TrackedToolBlock` still has required `index`, we need to handle this correctly.

The actual fix: The `delete` on line 308 runs for ALL block types (text, thinking, tool_use) before the `if/else if` block type dispatch. For `TrackedToolBlock`, `index` is also only needed during streaming. Make it optional too by changing the `TrackedToolBlock` interface in event-bridge.ts:

In `event-bridge.ts` line 24, change `TrackedToolBlock.index` from required to optional:
```typescript
interface TrackedToolBlock {
  type: "tool_use";
  index?: number;  // was: index: number
  id: string;
  name: string;
  claudeName: string;
  arguments: Record<string, unknown>;
  partialJson: string;
}
```

Then `delete block.index` works without cast for both `TrackedContentBlock` and `TrackedToolBlock` since both have optional `index`.

Audit the `blocks.findIndex((b) => b.index === event.index)` calls in event-bridge.ts -- these access `b.index` which is now optional. This is safe because `findIndex` checks `b.index === event.index` and if `b.index` is `undefined`, it simply won't match (correct behavior -- a block with deleted index should not match).

**Fix 4: event-bridge.ts line 336** -- `(contentBlock as any).arguments = finalArgs`

Per the research Deep Dive: `contentBlock` is already typed as `ToolCall`. `finalArgs` is `Record<string, unknown> | string`. `ToolCall.arguments` is `Record<string, any>`. Assigning `string` to `Record<string, any>` is the type mismatch.

Replace with `as Record<string, any>` on the right-hand side (more precise than `as any`, documents the intent that pi handles string arguments at runtime):

```typescript
// BEFORE:
(contentBlock as any).arguments = finalArgs;

// AFTER:
contentBlock.arguments = finalArgs as Record<string, any>;
```

This replaces `as any` on the left (which loses all type info on `contentBlock`) with `as Record<string, any>` on the right (which only broadens `finalArgs` to match the target type). The existing code comment at lines 338-340 already documents that pi handles string arguments at runtime.
